AI Product Portfolio

I bombed the AI PM questions.
So I built something.

After 10+ years in product, in 2025, I kept getting asked how I was using AI in my work. I didn't have a good answer. So I built two apps to find out what it takes to ship, the tradeoffs product teams face daily, and how AI is changing what's possible.

Read the 6-post series →
What I was trying to figure out

I kept asking myself: what does it actually feel like to build with AI tools when development cost is basically zero? Not just manage a team doing it, but write the specs, make the tradeoffs, and ship something real.

So I built two apps using Claude Code. Not to become a software engineer, but to build the intuition and vocabulary to be a better PM.

Honest questions I had
  • Where does AI actually need PM judgment, and where doesn't it? I wanted to find the line. Not read about it, but hit it myself and see what happened.
  • Do engineers still need a PM when AI can build anything? I had a real answer to this after building. I didn't before.
  • How much does an early decision actually cost you downstream? When you own the whole product, every tradeoff you deferred eventually shows up on your doorstep.
  • What do engineers think about that PMs never ask? Building database policies and wiring up auth flows gave me a different set of questions to bring to engineering conversations.
What I actually learned
01: Acceleration ≠ Automation

AI tools make you faster, not autonomous

Claude Code moved fast, but every meaningful decision still needed my judgment. Vague requirements produce vague code: writing a clear spec for AI is the same skill as writing one for an engineer.

02: The Bottleneck Shifts

PM judgment, not engineering velocity, is the constraint

When AI handles implementation, the bottleneck shifts to product clarity: what to build, in what order, for whom. Good product thinking gets more valuable as AI tools mature, not less.

03: Technical Fluency Changes Conversations

Shared vocabulary with engineers changes what's possible

Having built RLS policies, managed API costs, and wired up multi-step state myself, I now know which engineering tradeoffs are real and which are negotiable. That changes how I scope, estimate, and push back.

04: Prompting is Requirement Writing

Writing a prompt is the same work as writing a requirement

Writing the health analysis prompt (output format, tone, edge cases, fallbacks) felt exactly like writing a PRD. PMs who thrive in AI will be the ones who can translate user needs directly into model instructions.

05: Shipping Teaches What Planning Can't

Real usage surfaces gaps no backlog anticipates

Every sprint, fixing one thing exposed the next gap. Cutting scope, resisting over-engineering, shipping anyway: you don't really learn that until you own the deadline.

06: AI Creates New UX Problems

Latency, uncertainty, and hallucination are product problems

Healthy Pathways had latency, partial outputs, and model uncertainty, all displayed to health-conscious users. None of that is an engineering problem. It's a product design problem, and spinners don't solve it.

Healthy Pathways
Most people go to the doctor's office not knowing what to ask. Healthy Pathways surfaces what matters so you walk in with better questions.
Next.js TypeScript Tailwind Supabase Claude Sonnet Claude Haiku shadcn/ui
Live Demo GitHub
The problem

People are unclear about which health areas to focus on or which tests to ask their doctor for. Healthy Pathways uses your demographics, lifestyle, and lab results to surface what matters most, not to replace your doctor, but to help you walk in to your visit with better questions.

Key Features
  • 3-step assessment wizard with auto-save and resume; users never lose progress
  • AI-driven concern recommendations based on age, ethnicity, sex, and lifestyle
  • Progressive biomarker entry, grouped by concern and collapsed by default so users aren't overwhelmed
  • Traffic-light health review (Act Now / Monitor / Explore) with findings, actions, and doctor questions
  • Guest mode: rules-based analysis, no account needed, nothing stored server-side
  • Quality meter that tells users when they've entered enough for a meaningful analysis
  • Assessment history with PDF export, designed to be taken to a doctor's appointment
What I learned
  • Prompting is product design. The AI output is only as good as how you structure the input: edge cases, output format, tone. It's requirement writing.
  • LLM latency is a real UX constraint. A 10-second API call changes how you design the whole experience. I use Haiku for speed and Sonnet for depth, depending on what the user needs.
  • Invent & simplify. It's easy to keep adding. The harder call is cutting features that complicate more than they help.
  • The backlog never empties. Every fix revealed the next gap. Prioritizing under constraint is the same work whether AI or engineers are writing the code.

MB Connect
Scouts can't find which merit badge classes their troop is offering. Counselors don't know what scouts actually want. MB Connect connects them.
Next.js TypeScript Tailwind Supabase PostgreSQL RLS
Live Demo GitHub
The problem

Scouts struggle to find which Merit Badge classes their troop is offering. Counselors have no visibility into what badges scouts actually want. MB Connect bridges that gap: scouts can discover and enroll in classes, counselors can see demand and manage their offerings.

Key Features
  • Public merit badge catalog with search, category filters, and difficulty indicators, visible without an account
  • Role-based access for four user types (Scout, Counselor, Leader, Admin), each with their own permissions and views
  • Counselor qualification profiles: self-declared teaching credentials scoped to specific merit badges
  • Class creation and enrollment flow: counselors publish sessions, scouts discover and register
  • Merit badge detail pages listing available counselors, closing the loop from discovery to enrollment
  • Troop roster management for admins with role and join-date visibility
  • Full guest → account onboarding flow with invite-based troop joining
What I learned
  • Building for four user types at once is hard. Each persona has different goals, trust levels, and failure modes. Getting that right in the spec is the product work.
  • Row-level security is product policy in SQL. Every access decision (who can see what) is both a PM call and a database constraint. Get it wrong and it's a security bug, not just a UX issue.
  • Shipping revealed gaps specs couldn't predict. The onboarding "cold drop" (finishing setup with no clear next step) was invisible until the full flow existed.
  • Real data is messier than demo data. Seeding 135 actual BSA merit badges surfaced data quality issues I never would have found with a test dataset.
