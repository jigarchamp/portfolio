I'm a Technical Product Manager who learned by doing. These two apps were built solo using AI coding tools — to experience firsthand what it takes to ship a product, and to understand the decisions engineers make every day.

Why I built these

As a Product Manager at the intersection of AI and product development, I kept asking myself: what does it actually feel like to build with AI tools? Not just manage a team building them — but write the specs, make the tradeoffs, and ship something real.

So I built two full-stack web apps from scratch using Claude Code as my primary development tool. The goal wasn't to become a software engineer — it was to develop the intuition and vocabulary needed to be a more effective technical product leader.

Healthy Pathways
AI-powered personal health assessment tool. Enter your lab results and health profile — Claude AI generates a structured, actionable health review with traffic-light status indicators and doctor questions.
Next.js TypeScript Tailwind Supabase Claude Sonnet shadcn/ui
Live Demo GitHub
Key Features
  • 5-step guided assessment wizard with auto-save and resume — users never lose progress
  • Demographics-driven concern recommendations: AI surfaces what to prioritize based on age, sex, and lifestyle
  • Concern-grouped biomarker entry with progressive disclosure — reduces overwhelm on the test results step
  • Traffic-light health review (Act Now / Monitor / Explore) with findings, suggested actions, and doctor questions
  • Guest mode — 3-step flow, rules-based analysis, fully browser-side, no data sent to server
  • Analysis quality meter that signals when enough data has been entered to generate a meaningful review
  • Assessment history dashboard and Print / Save PDF support
Personal Learnings
  • Prompting is product design. The quality of the AI output is entirely determined by how you structure the input. Writing a good prompt is fundamentally PM work — defining requirements, handling edge cases, and specifying output format.
  • LLM latency is a UX constraint. A 10-second API call requires different UX patterns than a 200ms query. I designed loading states, fallback analysis, and a guest mode specifically around this constraint.
  • Wizard drop-off mirrors roadmap prioritization. Each step where users abandoned had a clear root cause — missing progress signals, ambiguous field labels, fear of losing data. The same diagnostic thinking applies to feature adoption.
  • Security is a product requirement, not an afterthought. Supabase RLS policies, API key management, and guest-vs-authenticated data handling required deliberate decisions that affected every part of the architecture.
  • The backlog never empties. At every sprint, fixing one experience revealed the next gap. Prioritization under constraint is the real work.

MB Connect
Merit badge management platform for BSA Scout troops. Scouts discover and track badges, counselors create and manage classes, troop leaders maintain roster visibility — all in one role-aware app.
Next.js TypeScript Tailwind Supabase PostgreSQL RLS
Live Demo GitHub
Key Features
  • Public merit badge catalog with search, category filters, and difficulty indicators — visible without an account
  • Role-based access control across four personas: Scout, Counselor, Leader, and Admin — each with distinct permissions and views
  • Counselor qualification profiles — self-declared teaching credentials scoped to specific merit badges
  • Class creation and enrollment flow — counselors publish sessions, scouts discover and register
  • MB detail pages showing available counselors for each badge, linking discovery to action
  • Troop roster management for admins with role and join-date visibility
  • Full guest → account onboarding flow with invite-based troop joining
Personal Learnings
  • Data modeling decisions define product constraints. Choosing between a JSONB column and a relational table for counselor qualifications locked in capabilities for future features. These are the decisions I now know to flag early.
  • Multi-persona products require genuine empathy for each user type. Designing for Scouts, Counselors, Leaders, and Admins simultaneously forced me to hold four different mental models — each with different goals, trust levels, and failure modes.
  • Row-level security is product policy expressed in SQL. Every access rule (can a leader see a counselor's qualifications?) is both a product decision and a database constraint. Getting these wrong is a security vulnerability, not just a UI bug.
  • Shipping alpha revealed gaps that specs could never predict. The onboarding "cold drop" (users completing setup with no next action) only became obvious after building the complete flow end-to-end.
  • Working with real-world data adds hidden complexity. Seeding 135 BSA merit badges with accurate categories, descriptions, and difficulty levels exposed data quality issues that a simple demo dataset never would.
Cross-project takeaways — building AI products vs. managing teams building them
01 — Acceleration ≠ Automation

AI tools make you faster, not autonomous

Claude Code dramatically accelerated development — but every significant decision still required judgment I had to bring. Ambiguous requirements produce ambiguous code. Writing precise specs for an AI assistant is the same skill as writing them for an engineering team.

02 — The Bottleneck Shifts

PM judgment, not engineering velocity, is the constraint

When AI handles implementation, the bottleneck becomes product clarity: what should be built, in what order, for whom? This experience reinforced that strong product thinking becomes more valuable as AI coding tools mature — not less.

03 — Technical Fluency Changes Conversations

Shared vocabulary with engineers changes what's possible

Having personally implemented Supabase RLS policies, API streaming, and multi-step wizard state management means I now know which tradeoffs are real and which are resolvable. That fluency changes how I scope work, interpret estimates, and push back on constraints.

04 — Prompt Engineering = Requirement Writing

AI product quality is determined at the spec layer

Designing the AI health analysis prompt for Healthy Pathways — defining output schema, tone, edge cases, and fallback behavior — was indistinguishable from writing product requirements. The best AI PMs will be those who can translate user needs into precise model instructions.

05 — Shipping Teaches What Planning Can't

Real usage surfaces gaps no backlog anticipates

Both apps went through multiple sprint cycles where each shipped improvement revealed the next critical gap. The discipline of ruthless prioritization, cutting scope to hit a milestone, and not gold-plating — that can only be learned by actually having a deadline.

06 — AI Requires New UX Paradigms

Latency, uncertainty, and hallucination are product problems

Building Healthy Pathways exposed UX challenges unique to AI: a 10-second wait for analysis, partial results, model uncertainty displayed to medical users. These aren't engineering problems — they're product design problems that require PM-led solutions, not just spinners.
